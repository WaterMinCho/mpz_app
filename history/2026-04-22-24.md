## deploy.yml 재설계

### 문제
worker 컨테이너 분리 시 deploy.yml을 `build` + `up -d --no-deps` 2단계로 분리 → 연쇄 문제 발생:
- frontend 빌드/실행 분리 → Docker가 새 이미지로 컨테이너 교체 안 함
- `--no-cache` 시도 → EC2 메모리 과부하 → 서버 다운 (SSH 불가)
- `CACHE_BUST` 추가 → `--force-recreate` 누락 → 컨테이너 교체 안 함
- 연속 push 시 배포 2개 동시 실행 → EC2 과부하

### 해결
- `up -d --build --force-recreate frontend backend nginx redis` **한 줄로 통합**
- `CACHE_BUST=$(date +%s)`를 `.env`에 넣어 docker-compose build arg로 전달
- `concurrency` 추가: 같은 브랜치 연속 push 시 이전 배포 자동 취소
- `sleep 5` 제거 → `until ... do sleep 2; done` 상태 기반 대기
- nginx 별도 restart 제거 (resolver가 자동 해결)
- `collectstatic --clear` 제거 (django-user 권한으로 root 파일 삭제 불가)

### 교훈
- 땜빵 수정이 연쇄 문제를 야기함. 단순한 구조가 안정적
- `sleep` 같은 시간 기반 대기는 예외/변수가 많아서 불안정 → 상태 기반 대기 사용
- EC2 t2.medium(4GB)에서 Docker 빌드 2개 동시 실행하면 메모리 과부하 → concurrency 필수

## Worker 컨테이너 분리

### 배경
- 동기화가 HTTP 요청으로 웹 서버(gunicorn) 안에서 실행 → 배포/재시작 시 같이 죽음
- 오늘 하루만 배포 때문에 동기화가 2번 끊김

### 구현
- docker-compose에 `worker` 서비스 추가 (같은 backend 이미지, 다른 command)
- `worker_entrypoint.sh`: migrate + `python manage.py run_scheduler`
- `run_scheduler` management command: Python 기반 cron (supercronic 실패로 대체)
- `sync_public_data` management command 개선: status_sync 전략, SyncLog 기록, --days 정상 동작
- deploy.yml: `--no-recreate worker`로 배포 시 worker 재시작 제외

### supercronic 실패
- Dockerfile에 supercronic 바이너리 추가 → slim-bullseye에서 `Failed to fork exec` 에러
- 원인: slim 이미지 호환성 문제
- 대안: Python 기반 스케줄러 (`run_scheduler` management command)로 대체

### 동기화 전략
| 전략 | 스케줄 | 조회 범위 | 이미지 처리 |
|------|--------|----------|------------|
| incremental | 매일 03:00 KST | 어제~오늘 | 다운로드 + 업로드 |
| status_sync | 매주 일 04:00 KST | 보호중 전체 | 스킵 (상태만) |
| full | 매월 1일 05:00 KST | 전체 (최근 90일) | 다운로드 + 업로드 |

### 교훈
- `docker compose exec -T`는 SSH 세션에 의존 → SSH 끊기면 프로세스도 종료
- `docker exec -d`로 detach 실행하면 SSH 무관하게 동작
- Worker 스케줄러 03:00 KST 자동 실행 확인됨

## 프론트엔드 개선

### breed + name 표시
- `getDisplayBreedName` 유틸 함수 (`lib/animal-utils.ts`)
- 종 있고 이름 있으면: "믹스견(깜돌이)"
- 종만 있으면: "믹스견"
- 이름만 있으면: "깜돌이"
- 둘 다 없으면: "종 미등록"
- 공공데이터 이름 `공공데이터_xxx`는 이름 없는 것으로 처리

### GPS 지역 판별
- **버그:** `findNearestRegion`에서 경기를 무조건 먼저 검사 (반경 90km) → 서울도 경기로 판별
- **수정:** 경기 우선 검사 제거 → 모든 지역 거리 순 비교
- 서울(반경 20km)이 경기(반경 90km)보다 가까우면 서울로 판별

### 홈 내 주변
- 진입 시 항상 "내 주변" 기본 선택 + GPS 자동 요청
- `isNearbyActive` 별도 상태로 내 주변/지역 버튼 하이라이팅 분리
- 지역 버튼 클릭 시 내 주변 해제
- "주목받고 있는 아이들" 섹션에서 region 필터 제거 (전체 확성기 순)
- **무한 루프 버그:** `requestLocation`을 useEffect dependency에 넣으면 매 렌더마다 새 함수 → 무한 루프 → `eslint-disable`로 마운트 1회만 실행
- **초기화 순서 에러:** `isNearbyActive` 선언이 useEffect보다 아래 → `Cannot access before initialization` → 선언 위치 이동

### 전문가 의견
- 상세 페이지 `SubscriberDetails` 주석 처리 (전문가 의견 + 전문가 한 마디)
- 필터 `expertOpinion` 비활성화
- **주의:** 센터 등록/수정의 `DetailInfo`는 별개 기능 — 건드리면 안 됨. "전문가 한 마디" textarea만 주석 처리

### 기타
- 배너 반응형: 고정 높이 → `aspect-ratio + max-h-[232px]`
- 커뮤니티/상담 플로팅 버튼 주석 처리
- 입양/찜 탭 sticky: `top-[54px]` (TopBar 높이)

- 로그인 페이지: `h-screen` → `100dvh` (모바일 Safari/Chrome)

- 카카오 프로필: `*.kakaocdn.net` 와일드카드 (img1, t1 등 서브도메인 대응)
